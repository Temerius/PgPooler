# Диплом: архитектура и готовность ядра

## Тема

**Программное средство управления пулом соединений и маршрутизации запросов в PostgreSQL.**

Аналоги: PgBouncer, Odyssey. Цель — пулинг соединений, маршрутизация по пользователю/БД, три режима сессий, конфиг (сейчас YAML, в дипломе можно описать и XML как вариант), Web GUI для мониторинга и управления (активные соединения, аналитика, kill/block), работа через Docker Compose (инструмент + GUI + БД аналитики + несколько Postgres для демо).

---

## Три типа сессий (режимы пула)

Как в PgBouncer и Odyssey:

| Режим | Когда соединение выдаётся | Когда возвращается в пул |
|-------|---------------------------|---------------------------|
| **Session** | При подключении клиента (один клиент — одно соединение с PG на всю сессию) | При отключении клиента |
| **Transaction** | На время транзакции (BEGIN … COMMIT/ROLLBACK) | После COMMIT/ROLLBACK (ReadyForQuery с состоянием idle) |
| **Statement** | На время одного запроса | После каждого завершённого запроса (каждый ReadyForQuery) |

В коде уже есть все три: `PoolMode::Session`, `PoolMode::Transaction`, `PoolMode::Statement` (config.hpp, client_session).

## Что уже сделано (ядро)

| Компонент | Статус | Примечание |
|-----------|--------|------------|
| Приём клиентов (TCP, PG Wire) | ✅ | Dispatcher + workers, libevent |
| Парсинг Startup (user, database) | ✅ | protocol/message |
| Маршрутизация по user/database | ✅ | routing.yaml: exact, list, prefix, regex, user, default |
| Несколько бэкендов | ✅ | backends.yaml, PoolManager |
| Пул соединений (take/put) | ✅ | BackendConnectionPool, ключ (backend, user, db) |
| Режимы Session / Transaction / Statement | ✅ | Логика в ClientSession (return после Z, по режиму) |
| Auth через бэкенд (proxy Startup) | ✅ | Одно соединение на auth, затем пул |
| DISCARD ALL при выдаче из пула | ✅ | SendingDiscardAll |
| Очередь ожидания (query_wait_timeout) | ✅ | ConnectionWaitQueue |
| Конфиг (YAML) | ✅ | pgpooler, logging, backends, routing — 4 файла |
| Docker Compose | ✅ | pgpooler + 2 Postgres (primary/replica), тесты маршрутизации |

**Итог:** ядро прокси + пул + маршрутизация + три режима сессий реализованы и работают. Для диплома этого достаточно как «ядро».

## Архитектура «аналитика + GUI»

Да, идея **«аналитическая БД + Python с GUI»** правильная и уже заложена в README:

- **Ядро (C++)** — только прокси, пул, маршрутизация. Либо встраивает простой REST API (статус, список сессий, kill/block), либо отдаёт метрики в очередь/файл, а отдельный сервис (Python) читает и пишет в Analytics DB.
- **Analytics DB (PostgreSQL)** — отдельный контейнер; таблицы: connection_log, active_connections, pool_stats_history, blocked_users, events. Нужна для истории и отчётов в GUI.
- **Web GUI (Python)** — Flask (или FastAPI), рендер страниц (Jinja2), запросы к REST API ядра (если есть) и к Analytics DB для графиков и истории. На порту 8080.

Варианты:

1. **REST API внутри ядра (C++)** — один процесс: и прокси, и HTTP на 8081. GUI дергает API ядра для «живых» данных (connections, kill, block) и ходит в Analytics DB для истории.
2. **REST API в Python** — ядро только прокси; отдельный Python-сервис читает логи/сокет/файл от ядра (или ядро шлёт метрики в БД), предоставляет REST и отдаёт данные GUI. Тогда «управление» (kill/block) либо через общую БД/файл, либо позже добавить в ядро минимальный HTTP.

Для диплома логично: **ядро + встроенный минимальный REST API (статус, список соединений, kill, block, reload)** и **Python GUI + Analytics DB**, как в README.

## Конфиг: XML vs YAML

В теме указано «правила в идеале прописать в XML». Сейчас всё в YAML. Варианты для диплома:

- **Вариант A:** Добавить загрузку правил из XML (pugixml): те же сущности (backends, routing, pool_mode), другой формат. YAML оставить как альтернативу.
- **Вариант B:** Оставить YAML, в тексте диплома обосновать выбор (читаемость, типично для таких систем), указать возможность поддержки XML по тому же описанию правил.

Оба варианта для защиты нормальны.

## Готовность к переходу на GUI и аналитику

**Кор (прокси + пул + маршрутизация + три режима) готов** для того, чтобы:

1. Описать в дипломе как реализованное ядро.
2. Строить дальше: REST API (в ядре или в Python), запись метрик в Analytics DB, Web GUI, полный Docker Compose.

Следующие логичные шаги:

1. **REST API** — хотя бы GET `/api/status`, GET `/api/connections`, POST `/api/connections/{id}/kill`, POST `/api/users/{name}/block` / `unblock`, GET `/api/pools`. Для этого в ядре нужен доступ к списку сессий и пулов (уже есть PoolManager, список ClientSession нужно либо хранить в одном месте, либо собирать по воркерам).
2. **Схема Analytics DB** — создать миграции/скрипты: connection_log, pool_stats_history, events, blocked_users.
3. **Сбор метрик** — из ядра в очередь или сразу в БД (отдельный поток или отдельный процесс на Python, читающий из ядра).
4. **Flask-приложение** — дашборд (активные соединения, пулы, бэкенды), страницы «соединения» (с кнопкой kill), «пользователи» (block/unblock), опционально графики из Analytics DB.
5. **Docker Compose** — сервисы: pgpooler, gui (Python), analytics-db (PostgreSQL), postgres (primary), postgres2 (replica).

Если нужно, следующий шаг можно расписать в виде конкретного плана по файлам и эндпоинтам (например, с чего начать: REST в ядре или схема БД + GUI без kill/block).
